## Key Implementation Details

### 1. Debounce Without a Library

`useBusinessDay` implements debounce with `useEffect` + `setTimeout(fn, 300)`. The cleanup function calls both `clearTimeout(timer)` and sets `cancelled = true`, preventing both the timer from firing and any in-flight fetch from updating state after the input changes or the component unmounts.

### 2. Client-Side Weekend Detection

Before starting the debounce timer, `getDay()` is checked (0=Sunday, 6=Saturday). If it's a weekend, the result is set immediately without an API call and the effect returns early. The `'T00:00:00'` suffix on the date string forces local-timezone parsing, preventing UTC midnight shifting the day-of-week in timezones west of UTC.

### 3. Year Range as Module-Level Constants

`CURRENT_YEAR` and `YEARS` are computed once at module load (not inside the component), so they never trigger re-renders and are stable across hot-reloads.

## Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Debounce implementation | `useEffect + setTimeout` | No extra dependency; straightforward and testable |
| Weekend check | Client-side before debounce | Instant feedback, avoids unnecessary API call |
| Date string parsing | `date + 'T00:00:00'` | Forces local timezone; `new Date('2026-05-23')` parses as UTC midnight and can shift day-of-week in negative-offset timezones |

## Developer Notes

- **Date parsing timezone trap**: `new Date('2026-05-23')` is UTC midnight → in UTC-5, `getDay()` returns Friday (May 22). Always append `'T00:00:00'` to get local midnight.
- **`CURRENT_YEAR` outside component**: Computing `new Date().getFullYear()` inside the component body runs on every render. Module-level constant is free.
- **`useBusinessDay` resets `result` to `null` when `date` is cleared** (the `if (!date || !currency)` guard) — the badge disappears cleanly when the input is cleared.
